## 1. Hybrid Cloud Burst Architecture & macOS vLLM Fixes

> **Type:** Infrastructure / Fix

Triển khai kiến trúc hybrid deployment: Docker Swarm local cluster là primary compute tier, tự động burst lên Google Cloud Run (serverless, min=0, max=20) khi CPU cluster vượt ngưỡng cấu hình (mặc định 70%).

Hai service mới điều phối logic burst:
- **Hybrid Router** (`hybrid-router/`) — FastAPI transparent reverse proxy, route request theo ngưỡng CPU hoặc Cloud Run.
- **Burst Controller** (`burst-controller/`) — Python daemon giám sát Prometheus và gọi `gcloud run services update`.

### Added
* `hybrid-router/main.py`, `requirements.txt`, `Dockerfile` — FastAPI reverse proxy với CPU-based routing và circuit breaker
* `burst-controller/controller.py`, `requirements.txt`, `Dockerfile` — Async CPU-monitoring daemon hysteresis-based scaling
* `docker-compose.swarm.yml` — Full Docker Swarm stack
* `scripts/setup-swarm.sh`, `scripts/burst-gcp.sh` — Setup scripts cho local Swarm và GCP

### Changed
* `.env.example` — Thêm Hybrid Cloud Burst section
* `monitoring/prometheus.yml` — Thêm node-exporter scrape targets, hybrid-router metrics, giảm scrape interval

### Removed
* Ray Serve dependency (replaced by Cloud Run autoscaling + Docker Swarm replica scaling)

### Fixed (macOS ARM64 vLLM)
* **Docker GPU Driver** — Xóa `deploy` block nvidia reservation trong `docker-compose.yml`; sửa lỗi `could not select device driver "nvidia"` trên macOS
* **vLLM Image** — Chuyển từ `vllm/vllm-openai:latest` sang `vllm/vllm-openai-cpu:latest-arm64`
* **CLI Args** — Xóa `--device cpu` không hỗ trợ, dùng positional `--model`; đổi dtype sang `bfloat16`, giảm `--max-model-len` xuống `1024`, thêm `--max-num-seqs 16` để tránh OOM Kill
* **vLLM CPU Autobind Crash** — Set `VLLM_CPU_OMP_THREADS_BIND="all"` để fix `AssertionError` libnuma không parse được Docker CPU topology trên macOS
* **NUMA Missing Metadata** — Thêm `VLLM_CPU_SIM_MULTI_NUMA=1` để simulate multi-NUMA topology

## 2. Replace vLLM with Ollama

> **Type:** Refactor

Thay thế vLLM Docker service bằng `ollama/ollama:latest`. Ollama dùng backend `llama.cpp` tối ưu cho Apple Silicon, tránh hoàn toàn các vấn đề NUMA assertions và OOM của vLLM CPU fallback.

Container Ollama tự động pull model `qwen2.5:3b` khi khởi động và alias thành `qwen3.5-plus` để tương thích 100% với backend hiện tại.

### Changed
* `docker-compose.yml` — Thay `ai_vllm` bằng `ai_ollama` (`ollama/ollama:latest`)
* `VLLM_BASE_URL` / `EMBEDDING_BASE_URL` — `http://vllm:8000` → `http://ollama:11434/v1`
* Volume — `hf_cache` → `ollama_data`
* Profile — `--profile vllm` → `--profile ollama`

## 3. Fix JSONDecodeError (Control Characters)

> **Type:** Fix

Sửa `JSONDecodeError` do unescaped control characters trong JSON payload tại LLM Gateway và vLLM proxy. Chuyển sang `json.loads(..., strict=False)` và thêm helper `get_safe_json`.

### Added
* `get_safe_json` helper trong gateway — decode body với `strict=False`
* Explicit string payload handling với fallback 400 errors thay vì 500 crashes

### Changed
* `gateway.py` — Đổi JSON extraction trong `chat_completions`, `embeddings`, `memchat_completions`
* `vllm_deployment.py` — Đổi body decoding logic

### Fixed
* `json.decoder.JSONDecodeError` khi `request.json()` gặp invalid bytes

## 4. Fix Gateway vLLM 404 Route Errors

> **Type:** Fix

Sửa `JSONDecodeError: Extra data` trong gateway routes. Nguyên nhân: `VLLM_BASE_URL` chứa trailing `/v1` dẫn đến endpoint sai (e.g. `/v1/v1/chat/completions`). Ollama trả về `404 page not found` dạng plain text, gateway parse JSON thất bại.

### Added
* Graceful handling của non-200 HTTP status trong `/v1/chat/completions` (regular + streaming) và `/v1/embeddings` trước khi decode JSON

### Changed
* Normalize `settings.vllm_base_url` khi build outbound HTTP POST request để tránh double `/v1/v1`

### Fixed
* JSON decode errors che giấu 404 / connection errors từ AI inference engines

## 6. Replace Ollama with llama.cpp Server

> **Type:** Refactor

Thay Ollama bằng **llama.cpp server** (`ghcr.io/ggerganov/llama.cpp:server`). llama.cpp nhẹ hơn Ollama, native GGUF support, không có native dependencies, OpenAI-compatible API. Một `model-downloader` init container tự fetch GGUF từ HuggingFace lần đầu và cache vào volume `models_data`.

### Added
* `model-downloader` init service — pull `Qwen2.5-3B-Instruct-Q4_K_M.gguf` từ HuggingFace; skip nếu đã có
* `llama-cpp` service — `ghcr.io/ggerganov/llama.cpp:server`, OpenAI-compatible API trên port 8080 (map host 11434)
* `models_data` Docker volume
* Env vars: `LLAMA_CPP_MODEL`, `LLAMA_CPP_CTX_SIZE`, `LLAMA_CPP_THREADS`
* Healthcheck trên `llama-cpp` service (`GET /health`)

### Changed
* `VLLM_BASE_URL` / `EMBEDDING_BASE_URL` — `http://ollama:11434/v1` → `http://llama-cpp:8080/v1`
* `.env.example` — LLM section đổi từ vLLM/Ollama sang llama.cpp fields
* `docs/architecture.md` — Cập nhật service inventory, LLM serving section, single-node diagram

### Removed
* `ollama` service (`ollama/ollama:latest`, profile `ollama`)
* `ollama_data` Docker volume
* Ollama-specific env vars (`VLLM_MODEL`, `VLLM_SERVED_NAME`, `VLLM_MAX_LEN`)
